# StartupOS — Website Rendering Plan

> Audit date: 2026-06-19
> Scope: Design the rendering layer that converts WebsiteSpec into a deployable website

---

## Question 1: What Information Exists in WebsiteSpec?

The WebsiteSpec is generated by AI from a Blueprint. Here is the exact structure:

```typescript
interface WebsiteSpecResult {
  pages: PageSpec[];
  theme: ThemeSpec;
  components: ComponentSpec[];
}

interface PageSpec {
  name: string;                    // "Home", "About", "Pricing"
  slug: string;                    // "/", "/about", "/pricing"
  sections: SectionSpec[];
}

interface SectionSpec {
  type: string;                    // "hero", "features", "services", "about", "cta"
  order: number;                   // 1, 2, 3 (determines vertical position)
  content: Record<string, unknown>; // freeform - varies by section type
}

interface ThemeSpec {
  primaryColor: string;            // "#1a1a2e"
  secondaryColor: string;          // "#16213e"
  fontFamily: string;              // "Inter"
  borderRadius: string;            // "8px"
}

interface ComponentSpec {
  name: string;                    // "Navbar", "Footer"
  type: string;                    // "navigation", "footer"
  props: Record<string, unknown>;  // freeform - varies by component
}
```

The AI prompt (in `provider.ts:129-151`) instructs the model to generate:
- Pages with sections like hero (`headline`, `subheadline`, `ctaText`), features (`items: [...]`)
- A theme with 2 colors, 1 font, 1 border radius
- Components like navbar and footer

The mock data (`mock-deliverables.json`) shows real-world examples:
- Sections: hero, services, features, about
- Content: freeform text strings or objects with title/content

## Question 2: Is WebsiteSpec Sufficient to Render a Complete Website?

**No.** WebsiteSpec is a design document, not a rendering specification. It describes WHAT to show, not HOW to build it.

### What WebsiteSpec CAN drive:
- Page structure (which pages exist, in what order)
- Section layout (vertical ordering of sections per page)
- Section content (headlines, subheadlines, feature lists, CTAs)
- Basic theming (2 colors, 1 font, border radius)

### What WebsiteSpec CANNOT drive:
- Actual HTML structure (no element types, no nesting)
- Responsive behavior (no breakpoints, no mobile variants)
- Images (no image URLs, no generation, no placeholder strategy)
- Layout (no column counts, no spacing, no grid)
- Navigation items (components are generic, no actual nav links)
- Footer content (no copyright text, no social links, no sitemap)
- SEO (no meta descriptions, no Open Graph, no structured data)
- Forms (no action URLs, no field definitions)
- Interactions (no hover states, no animations, no scroll behavior)
- Accessibility (no ARIA labels, no semantic HTML choices)

## Question 3: What Information Is Missing?

| Missing Information | Impact | How to Fill |
|---------------------|--------|-------------|
| **Images** | Hero sections need visuals. No image URLs in spec. | Use Unsplash API for relevant photos, or CSS gradients/patterns as fallback |
| **Layout structure** | Section type is a string, not a layout descriptor. "hero" could be center-aligned, left-aligned, split, etc. | Define layout templates per section type. AI generates section TYPE, renderer picks LAYOUT |
| **Responsive design** | No mobile/tablet content. | Renderer applies responsive CSS defaults (single column on mobile, multi-column on desktop) |
| **Navigation items** | ComponentSpec has no nav links. | Derive nav from page slugs. Navbar items = list of pages with name + slug |
| **Footer content** | No copyright, links, social. | Renderer generates footer with: copyright year, startup name, derived nav links |
| **SEO metadata** | No title, description, OG tags. | Renderer generates meta from: page name + blueprint description + startup name |
| **Actual copy** | Section content is `Record<string, unknown>` — may be empty or inconsistent. | Renderer uses fallback content when fields are missing. "headline not set" → generate placeholder |
| **CTA destinations** | Hero CTA text exists, but no link URL. | Derive: primary CTA → "/contact" or "/pricing". Secondary CTA → "/about" |
| **Contact info** | No email, phone, address. | Renderer generates contact section with placeholder or omit if not in spec |
| **Social links** | No Twitter, LinkedIn, GitHub URLs. | Omit social links in MVP. Add later if spec includes them |

## Question 4: What Rendering Architecture Should Be Used?

### Architecture Options Evaluated

| Option | Build Step | Deploy Speed | Complexity | Quality | Verdict |
|--------|-----------|-------------|------------|---------|---------|
| **Static HTML** | None | Instant | Low | Medium | **Best for MVP** |
| **React SPA** | Vite build (~30s) | Fast | Medium | High | Good for v2 |
| **Next.js** | npm install + build (~2min) | Slow | High | Highest | Overkill |
| **Astro** | npm install + build (~1min) | Medium | Medium | High | Good for v2 |

### Recommendation: Static HTML Generation

**Why static HTML is the right choice:**

1. **No build toolchain** — No npm install, no Vite, no webpack. Generate files directly.
2. **Instant deployment** — Files are ready the moment the renderer finishes. No 30-120s build wait.
3. **Universal hosting** — Works on Vercel, Netlify, Cloudflare Pages, S3, any web server.
4. **Most reliable** — Fewer moving parts = fewer failure modes in the AI → render → deploy pipeline.
5. **Already designed for** — The `WebsiteResult` type in `types/ai.ts` already expects HTML/CSS/JS strings per page.
6. **SEO-friendly** — Server-rendered HTML is indexable by default. No client-side rendering issues.
7. **Fast to build** — A template-based renderer can be implemented in 3-5 days.

**Tradeoff:** Static HTML means no client-side interactivity beyond CSS transitions. This is acceptable for marketing/landing pages (StartupOS's target use case).

## Recommended Rendering Architecture

### Pipeline

```
WebsiteSpec (JSON)
    ↓
[Rendering Engine]
    ↓
WebsiteResult (HTML + CSS + JS per page + assets)
    ↓
[Deployment] → Static hosting → Public URL
```

### Rendering Engine Design

```
src/services/renderer/
├── index.ts              # renderWebsite(spec) → WebsiteResult
├── templates/
│   ├── base.html         # HTML shell with {{head}}, {{body}}, {{scripts}}
│   └── styles.css        # Base responsive CSS + theme variables
├── sections/
│   ├── hero.ts           # Renders hero section from content
│   ├── features.ts       # Renders features grid
│   ├── services.ts       # Renders services list
│   ├── about.ts          # Renders about section
│   ├── cta.ts            # Renders call-to-action
│   └── index.ts          # Section registry (type → renderer)
├── components/
│   ├── navbar.ts         # Renders navigation from page list
│   └── footer.ts         # Renders footer with derived links
├── theme.ts              # Injects colors/font/radius into CSS variables
├── meta.ts               # Generates <head> tags (title, description, OG)
└── assets.ts             # Generates placeholder images (CSS gradients/SVG)
```

### Section Rendering

Each section type has a dedicated renderer. The section type string maps to a renderer function:

```typescript
// Section registry
const sectionRenderers: Record<string, (content: Record<string, unknown>, theme: ThemeSpec) => string> = {
  hero: renderHeroSection,
  features: renderFeaturesSection,
  services: renderServicesSection,
  about: renderAboutSection,
  cta: renderCTASection,
};
```

Each renderer:
1. Takes the section's `content` object (freeform JSON)
2. Applies the theme (colors, font, radius)
3. Returns an HTML string
4. Handles missing content with sensible defaults

### Page Assembly

For each page in `WebsiteSpec.pages`:
1. Render navbar (derived from page list)
2. Render each section in order (sorted by `order` field)
3. Render footer
4. Wrap in base HTML template
5. Inject theme CSS variables
6. Generate meta tags from blueprint + page name
7. Return `{ name, slug, html, css, js }`

### Theme Application

```css
/* Generated CSS variables from ThemeSpec */
:root {
  --primary: #1a1a2e;
  --secondary: #16213e;
  --font: 'Inter', sans-serif;
  --radius: 8px;
}
```

Theme is injected as CSS custom properties. All section renderers use these variables.

### Placeholder Images

Since WebsiteSpec has no image references:
- **Hero sections**: CSS gradient backgrounds using theme colors
- **Feature icons**: Inline SVG or emoji
- **Background patterns**: CSS-only patterns (dots, waves, gradients)
- **No external image dependencies** — keeps rendering fast and reliable

## What NOT to Generate

| Item | Reason |
|------|--------|
| React components | No build step available. Would need Vite/Webpack integration. |
| TypeScript files | No compilation step. Static HTML works immediately. |
| Node.js server | Static files don't need a server. Hosting platforms serve them. |
| Package.json / dependencies | No npm install step. Files are self-contained. |
| Build configuration | No build needed. Output is final. |

## Integration with Existing Code

### Where rendering fits in the pipeline

The current worker flow (`src/queue/worker.ts:105-184`):

```
1. Read blueprint from DB
2. Generate WebsiteSpec via AI ← EXISTS
3. Store spec in WebsiteSpec table ← EXISTS
4. Create Website record with status "spec_generated" ← EXISTS
```

**The gap:** Step 4 stores `content: {}`. The rendering step should fill `Website.content` with the rendered output.

### Modified flow

```
1. Read blueprint from DB
2. Generate WebsiteSpec via AI
3. Store spec in WebsiteSpec table
4. RENDER: Convert WebsiteSpec → HTML/CSS/JS per page  ← NEW STEP
5. Store rendered output in Website.content (as JSON with pages array)
6. Update Website.status to "rendered"
```

### Website.content schema (proposed)

```typescript
// Website.content JSON structure after rendering
{
  "pages": [
    {
      "name": "Home",
      "slug": "/",
      "html": "<!DOCTYPE html>...",
      "css": ":root { --primary: #1a1a2e; } ...",
      "js": ""
    },
    {
      "name": "About",
      "slug": "/about",
      "html": "...",
      "css": "...",
      "js": ""
    }
  ],
  "assets": [
    {
      "path": "favicon.svg",
      "content": "<svg>..."
    }
  ]
}
```

This matches the existing `WebsiteResult` type defined in `types/ai.ts:52-67`.

### Deployment connection

The deployment worker (`src/queue/worker.ts:187-258`) currently just writes a URL. With rendered content in `Website.content`, the deployment step can:

1. Read `Website.content` (contains HTML/CSS/JS per page)
2. Upload files to hosting platform (Vercel/Netlify API)
3. Store the real deployment URL from the hosting platform response

## Answer to the Core Question

> Should StartupOS generate WebsiteSpec JSON or actual websites?

**Both, in sequence.**

1. **WebsiteSpec** is the AI's job — it's good at generating structured design decisions (page layout, section types, content, theme). This already works.

2. **Rendering** is the engine's job — it takes the spec and produces actual HTML/CSS/JS. This is a deterministic template system, not an AI task. It should be built as a separate service.

3. **The spec is NOT sufficient alone.** The renderer needs to fill gaps (images, layout, responsive, SEO, navigation) using templates and defaults.

4. **Static HTML is the fastest path** to a working website. React/Next.js can be added later for interactive sites, but static HTML covers 80% of marketing/landing page use cases.

**The fastest path to a working website:**

```
Founder clicks "Generate Website"
    ↓
AI generates WebsiteSpec (already works)
    ↓
Renderer converts spec → static HTML (3-5 days to build)
    ↓
Deployer uploads to Vercel (3-5 days to build)
    ↓
Vercel returns real URL
    ↓
Founder gets public URL that actually works
```

**Estimated total: 7-11 days of focused development.**
